# 突破程序员圈层：AI Skill 的真正普及问题

**Date:** 2026-05-08

---

## 你说对了什么

在写这个分析之前，先承认你的四个观察都是准确的：

1. **Skill 生态是程序员圈子的自嗨** — SKILL.md、CLI、GitHub repo、YAML frontmatter，这些对 99% 的人毫无意义
2. **信息已经爆炸到正常人无法跟踪** — skills.sh 上 91,000+ 个 skill，没人能评估
3. **大多数人不需要那么多 skill** — 他们需要的是 3-5 个场景，不是 35 个技能文件
4. **Chatbox 是瓶颈，不是未来** — 大多数人不想"跟 AI 对话"，他们想"把事做了"

这些问题本质上指向同一个核心矛盾：**当前的 AI 工具是为建造者设计的，不是为使用者设计的。**

## 1. 问题诊断

### 当前生态的真实用户画像

```
当前 AI Skill 用户 = 开发者 + 技术爱好者
  ├── 会用命令行
  ├── 理解 YAML
  ├── 有 GitHub 账号
  ├── 能阅读和理解 SKILL.md
  ├── 享受配置和定制的过程
  └── 约占 AI 用户的 5%

真正的大众用户 = 其他 95%
  ├── 不知道什么是 CLI
  ├── 不理解 YAML/JSON
  ├── 没有 GitHub 账号
  ├── 不想"配置"任何东西
  ├── 只想解决手头的问题
  └── 对他们来说，AI = 微信里的机器人 / Siri / ChatGPT 网页版
```

### 为什么走不出去

| 障碍 | 具体表现 |
|------|---------|
| **入口门槛** | 需要安装 CLI、配置环境、理解 Skill 加载机制 |
| **抽象层次** | Skill 是给 AI 的指令，不是给人用的界面 |
| **发现成本** | 91,000+ 个 Skill，普通人怎么选？选错了怎么办？ |
| **信任问题** | 第三方 Skill 可能包含恶意代码，普通人无法判断 |
| **价值不直观** | "这个 Skill 让 AI 更好地做 code review" — 非 programmers 不做 code review |
| **维护负担** | Skill 需要更新、调试、适配 — 谁来做？ |

### McKinsey 的数据证实了这一点

> **88% 的组织在用 AI，但只有 6% 实现了企业级影响。**
> 
> 从"试试 AI"到"受益于 AI"之间的鸿沟巨大。

Source: [McKinsey — Closing the Gap](https://www.lootzysoft.com/blog/the-state-of-ai-in-2025-closing-the-gap-between-adoption-and-impact)

## 2. 重新定义：从 "Skill" 到 "场景"

### 核心洞察

**人们不买"技能"，他们买"解决方案"。**

没有人会说"我需要一个 test-case-generator skill"。他们说：

- "我下周要测试这个功能，帮我准备测试用例"
- "这个 bug 怎么复现？"
- "帮我把这周的测试结果整理成报告"

**Skill 是给 AI 的指令。"场景"是给人的体验。** 好的产品把 Skill 藏在场景背后，用户完全不需要知道 Skill 的存在。

### 类比

| 用户要的 | 不是 | 而是 |
|---------|------|------|
| "帮我写周报" | Skill: weekly-report-generator | 一个"周报"按钮，点击后自动生成 |
| "这个 bug 是什么优先级？" | Skill: bug-triage | 把 Issue 丢进去，自动返回标签和优先级 |
| "我有个新产品想法" | Skill: brainstorming → prd-writer | 一个"从想法到 PRD"的向导流程 |
| "帮我做 PPT" | Skill: pptx | 上传内容 → 选模板 → 下载 PPT |
| "把这段话翻译成英文" | Skill: translate | 右键菜单 → 翻译 |

### 映射关系

```
用户看到的：场景（一个按钮、一个表单、一个流程）
           ↓ 背后映射
系统看到的：Skill（SKILL.md 文件）
           ↓ 实际执行
AI 看到的：指令（Skill 的 content 部分）
```

**用户永远不需要知道 Skill 的存在。** 他们只需要看到场景，点击，获得结果。

## 3. 跳出 Chatbox：四种交互范式

### 当前的问题

Chatbox 的本质是：**用户需要主动向 AI 描述他们想要什么。**

这对开发者没问题——开发者习惯精确描述需求。但大多数人不擅长"提示工程"。他们不知道怎么描述想要什么，也不知道 AI 能做什么。

**Chatbox 把使用 AI 的门槛放在了"你会不会提问"上。这是错误的门槛。**

### 四种更好的交互范式

#### 范式 1：按钮驱动（One-Click）

用户不需要打字，只需要点按钮。

```
┌─────────────────────────────────────┐
│  今天的 QA 工作台                    │
│                                      │
│  ┌──────────┐  ┌──────────┐         │
│  │ 🐛 待处理 │  │ 📝 新功能 │         │
│  │   3 个    │  │  待测试   │         │
│  │  [一键处理]│  │  [生成用例]│         │
│  └──────────┘  └──────────┘         │
│                                      │
│  ┌──────────┐  ┌──────────┐         │
│  │ 📊 本周   │  │ ⚡ 回归   │         │
│  │  测试报告 │  │  测试     │         │
│  │  [生成]   │  │  [运行]   │         │
│  └──────────┘  └──────────┘         │
└─────────────────────────────────────┘
```

用户点击"一键处理"→ 后台自动调用 bug-triage skill → 显示结果。

**Skill 在哪？** 藏在按钮后面。用户不知道、不需要知道。

#### 范式 2：模板驱动（Template）

用户填表，AI 填充细节。

```
┌─────────────────────────────────────┐
│  创建测试用例                         │
│                                      │
│  功能名称：[用户注册              ]   │
│  功能描述：[用户用邮箱密码注册，    ]   │
│             [24小时内需验证邮箱    ]   │
│                                      │
│  测试类型：☑ 功能  ☑ 边界  ☑ 异常   │
│  优先级：  ○ 基本  ● 完整           │
│                                      │
│  [生成测试用例]                       │
└─────────────────────────────────────┘
```

用户填了几行 → 后台调用 test-case-generator skill → 返回 15 个测试用例。

**Skill 在哪？** 藏在模板后面。

#### 范式 3：流程驱动（Flow）

用户跟着步骤走，不需要知道整体逻辑。

```
┌─────────────────────────────────────┐
│  🚀 新功能上线检查                    │
│                                      │
│  Step 1/5: 需求确认                   │
│  ┌─────────────────────────────────┐ │
│  │ 功能：用户注册                   │ │
│  │ PRD 链接：[粘贴链接或上传文件]   │ │
│  │ 预计上线：2026-05-15            │ │
│  └─────────────────────────────────┘ │
│                                      │
│  [下一步：生成测试用例 →]             │
│                                      │
│  进度：●○○○○                         │
└─────────────────────────────────────┘
```

后面是 5 个步骤：需求确认 → 测试用例 → 开发 → 测试执行 → 上线确认。每个步骤背后是一个或多个 Skill。

**Skill 在哪？** 藏在流程的每一步后面。

#### 范式 4：被动触发（Automatic）

用户什么都不用做。AI 主动工作。

```
┌─────────────────────────────────────┐
│  🔔 通知                             │
│                                      │
│  GitHub 有 3 个新 Issue：            │
│  ✅ #142 — 已自动分类为 P2/UI        │
│  ✅ #143 — 已自动分类为 P0/API       │
│  ⚠️ #144 — 需要你确认（可能是功能请求）│
│                                      │
│  [查看详情]  [全部确认]               │
└─────────────────────────────────────┘
```

Webhook 触发 → 自动调用 bug-triage skill → 只有不确定的才推给用户确认。

**Skill 在哪？** 完全透明。用户只看到结果。

### 四种范式对比

| 范式 | 用户操作 | Skill 可见性 | 适用场景 |
|------|---------|-------------|---------|
| **按钮** | 点一下 | 隐藏 | 重复性操作（生成报告、运行测试） |
| **模板** | 填表单 | 隐藏 | 结构化输入（写测试用例、提 Bug） |
| **流程** | 跟着走 | 隐藏 | 多步骤工作流（上线检查、新人入职） |
| **自动** | 无 | 完全透明 | 被动任务（Bug 分诊、监控报警） |

**共同点：用户永远不直接接触 Skill。** Skill 是实现细节，不是产品功能。

## 4. 信息过载的解法：不是更多选择，是更少选择

### 问题

skills.sh 有 91,000+ 个 Skill。这是 npm 问题的 10 倍。没有人能从这个数量中做出选择。

而且大多数 Skill 质量堪忧——正如你之前的研究发现的，Skillmark 评分最高的也只有 16.3%。

### 错误的解法

- ❌ 更好的搜索引擎（用户不知道该搜什么）
- ❌ 更多的筛选器（用户不知道该筛什么）
- ❌ 更详细的评分系统（用户不信任评分）

### 正确的解法：策展（Curation）

**不给用户选择。给他们答案。**

#### 层级 1：场景模板库（20 个，不是 91,000 个）

```
你是一个 QA 工程师？这里是你需要的 5 个场景：
  1. Bug 分诊       → 一键分类新 Bug
  2. 测试用例生成    → 从 PRD 生成测试
  3. 回归测试       → 运行并跟踪结果
  4. 测试报告       → 自动生成周报
  5. 上线检查       → 多步骤检查流程

你是一个产品经理？这里是你需要的 5 个场景：
  1. PRD 生成       → 从想法到文档
  2. 竞品分析       → 自动收集分析
  3. 需求优先级     → 智能排序
  4. 会议纪要       → 自动整理
  5. 进度追踪       → 跨项目汇总

你是一个设计师？这里是你需要的 5 个场景：
  1. 设计规范       → 自动生成 Design System
  2. 配色方案       → 从品牌色生成色板
  3. 组件文档       → 自动生成组件使用说明
  4. 设计走查       → 检查设计一致性
  5. 交付文件       → 自动标注和切图
```

**每个场景背后可能用了 3-5 个 Skill，但用户只看到 1 个按钮。**

#### 层级 2：角色定制包（预配置）

用户注册时选择角色 → 自动获得对应的场景包 → 不需要安装任何东西。

```
注册时：
  "你的角色是？"
  ○ 开发者  ○ QA  ○ 产品经理  ○ 设计师  ○ 运营  ○ 学生

选择"QA"后：
  → 自动获得 5 个场景
  → 预配置好 Bug 分诊、测试生成、报告等
  → 连接到你的 GitHub/Jira
  → 明天上班就可以用
```

#### 层级 3：渐进式展开

用户用了一个场景觉得好 → 系统推荐相关的 1-2 个场景 → 自然扩展。

```
用户用了"Bug 分诊"3 次后：
  "你可能也会喜欢：自动生成测试报告"
  [试试看]  [不用了]

用户点了"试试看"：
  → 新场景自动添加
  → 无需安装、配置、理解 Skill
```

### 策展的本质

**Apple Music 的播放列表 vs Spotify 的搜索栏。**

- Spotify 给你 1 亿首歌，让你自己找（技能市场）
- Apple Music 给你 30 个播放列表，由编辑精选（策展）

普通人选择 Apple Music 的方式。他们不需要 91,000 个 Skill。他们需要 5 个场景。

## 5. 推广路径：从程序员到普通人

### 三阶段渗透模型

```
阶段 1（现在）     阶段 2（6-12月）    阶段 3（12-24月）
面向开发者         面向技术团队         面向所有人
    │                   │                   │
    ▼                   ▼                   ▼
 CLI + SKILL.md    团队 Web 应用        场景化产品
 npx skills add    一键配置场景          角色模板包
 GitHub 生态       多人共享 AI 记忆      按钮驱动
 程序员自建        团队级 Skill 池       无需理解 AI
    │                   │                   │
    ▼                   ▼                   ▼
  100K 用户         1M 用户              10M+ 用户
 开发者社区         科技公司/创业团队      所有职场人士
```

### 阶段 1：当前（面向开发者）

**已经在做的事。** 继续做，但调整叙事：

- 不说"我们有 35 个 Skills"
- 说"我们解决了开发者的 5 个核心痛点"
- 每个"痛点"是一个完整的体验，不是一个 Skill 列表

### 阶段 2：团队级产品

**做一个团队版。** 不是让每个人自己配 Skill，而是：

```
团队管理员：
  1. 选择团队角色（开发、QA、设计、PM）
  2. 连接工具（GitHub、飞书、Notion）
  3. 选择场景（从 20 个预配置场景中选）
  
团队成员：
  1. 登录 → 看到自己的工作台
  2. 点击按钮 → AI 帮忙做事
  3. 不需要知道什么是 Skill
```

**关键变化：** 从"每个人自己配"到"管理员配一次，全团队用"。

### 阶段 3：大众产品

**做场景化的垂直产品。** 不是"AI Skill 平台"，而是：

- "QA AI 助手" — 给 QA 工程师的一键工具
- "PM AI 助手" — 给产品经理的一键工具
- "设计 AI 助手" — 给设计师的一键工具

每个产品是独立的品牌，独立推广，独立获客。背后共享同一个 Skill 引擎。

### 推广渠道的变化

| 阶段 | 渠道 | 信息 |
|------|------|------|
| 1 开发者 | GitHub、Hacker News、Reddit | "Extend your AI coding workflow" |
| 2 团队 | Product Hunt、技术博客、企业 BD | "给团队一个 AI 工作台" |
| 3 大众 | 小红书、抖音、微信公众号 | "一键搞定你的 XXX 工作" |

**注意阶段 3 的信息变化：** 不是"AI 能力"的推广，是"解决具体问题"的推广。

- ❌ "我们的 AI 可以帮你做很多事情"
- ✅ "一键生成 QA 报告，5 分钟搞定原来 2 小时的工作"
- ❌ "支持 91,000 个 Skills"
- ✅ "专为 QA 工程师设计的 5 个工具"

## 6. 产品形态：不是 Skill 平台，是 Workflow 产品

### 最终结论

**不应该做 "Skill 平台"。应该做 "场景化 Workflow 产品"。**

### 架构

```
底层：Skill 引擎（你已有的 SKILL.md 生态）
  ├── 系统 Skills：开发工作流
  ├── 领域 Skills：QA、PM、设计、运营
  └── 自定义 Skills：用户通过使用自动生成

中层：场景映射层
  ├── 每个"按钮"映射到 1-N 个 Skill
  ├── 每个"模板"映射到 Skill 的输入格式
  ├── 每个"流程"映射到 Skill 的执行顺序
  └── 每个"自动化"映射到 Skill + 定时触发

上层：产品界面
  ├── Web 版：场景化工作台
  ├── 桌面版：Tauri 应用（已有 desktop-app skill）
  ├── 移动版：微信/飞书小程序（触达大众）
  └── API 版：给其他产品集成
```

### 你的差异化

| 维度 | skills.sh / ClawHub | Kollab | 你的产品 |
|------|-------------------|--------|---------|
| **面向** | 开发者 | 团队 | 先开发者，后团队，最终大众 |
| **形态** | Skill 目录 | AI Workspace | 场景化 Workflow 产品 |
| **技术** | SKILL.md | 私有格式 | SKILL.md（开放标准） |
| **模型** | 任意 | 绑定 | 任意（ai-config） |
| **平台** | CLI/Web | Web | CLI/Web/桌面/移动 |
| **门槛** | 高（需懂技术） | 中（需理解团队概念） | 低（选角色 → 点按钮） |
| **数量** | 91K+ Skills | 少量内置 | 20 个场景（背后 N 个 Skills） |

### 第一步做什么

**不要做平台。做一个垂直场景产品。**

选一个你最熟悉的场景（比如 QA），做一个小而美的产品：

```
产品名：QA Copilot（或更好的名字）
一句话：AI 帮 QA 工程师处理日常重复性工作

功能（只有 5 个按钮）：
  1. Bug 分诊 → 自动分类新 Bug
  2. 测试生成 → 从 PRD 生成测试用例
  3. 回归执行 → 跟踪测试结果
  4. 报告生成 → 一键生成 QA 周报
  5. 上线检查 → 多步骤发布确认

技术：innate-frontend + Tauri（桌面版）或 Web
背后：bug-triage + test-case-generator + qa-report-generator + flaky-test-detector Skills

获客：QA 社区、测试论坛、小红书"QA 效率工具"
```

**一个产品做透一个场景，比一个平台覆盖所有场景更有穿透力。**

做完 QA 场景 → 做 PM 场景 → 做设计场景 → 自然形成平台。

但你从第一天就只说"AI QA 工具"，不说"AI Skill 平台"。因为没有人想买一个"平台"。人们买的是解决问题的工具。
