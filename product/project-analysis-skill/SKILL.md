# Project Analysis & Design Skill

> 从原始需求文档生成技术分析、架构设计、功能分解和实施计划等多维度文档的 Skill

## 用途

本 Skill 用于将简单的原始需求文档（如产品想法、用户需求描述）转化为完整的项目设计文档集，包括可行性分析、技术选型、架构设计、功能分解和实施计划。

## 输入

原始需求文档，包含以下内容：
- 项目目标/愿景
- 目标用户群体
- 核心功能需求
- 技术要求/约束
- 期望的输出形式

示例输入格式参考：`tasks/mindstorm/local-ai-playgound.md`

## 输出文档集

执行本 Skill 将生成以下文档：

```
tasks/
└── [project-name]/
    ├── analysis.md          # 技术分析与可行性评估
    ├── architecture.md      # 系统架构设计
    ├── features/            # 功能分解文档
    │   ├── F1-xxx.md
    │   ├── F2-xxx.md
    │   └── ...
    ├── plan.md              # 实施计划
    └── assets/              # 设计资源
        └── diagrams/        # 示意图文件
```

## 执行步骤

### Step 1: 需求理解与分析

阅读原始需求文档，提取关键信息：

| 维度 | 提取内容 |
|------|----------|
| **用户群体** | 目标用户的技能水平、使用场景、痛点 |
| **核心功能** | 必须实现的功能清单 |
| **技术约束** | 平台要求（Web/桌面/移动）、性能要求、安全要求 |
| **成功标准** | 如何衡量项目成功 |

**输出**: `analysis.md` 的第一部分「项目概述」

---

对比不同技术方案，给出推荐：

```markdown

|------|------|------|--------|
| [技术名] | [版本] | [用途] | ⭐⭐⭐⭐⭐ |

### 备选方案对比

| 对比项 | 方案A | 方案B | 结论 |
|--------|-------|-------|------|
| 安装包大小 | ~10MB | ~200MB | 方案A更优 |
| ... | ... | ... | ... |
```

**输出**: `analysis.md` 的第二部分「技术选型分析」

## 最佳实践

### 原始需求文档撰写指南

为了让本 Skill 发挥最佳效果，原始需求文档应包含：

1. **清晰的用户画像**
   - 技能水平（小白/初级/中级）
   - 使用场景
   - 痛点描述

2. **明确的功能边界**
   - 必须有的功能（MVP）
   -  nice-to-have 功能
   - 明确排除的功能

3. **技术约束**
   - 目标平台（Web/桌面/移动端）
   - 技术偏好（如果有）
   - 性能/安全要求

4. **期望输出形式**
   - 文档格式
   - 是否需要示意图
   - 详细程度要求

### 输出文档质量检查清单

- [ ] 每个架构图都有对应的文字说明
- [ ] 功能分解的粒度适中（每个功能2-5天工作量）
- [ ] 技术选型有明确的对比和理由
- [ ] 难点分析包含具体解决方案
- [ ] 计划中有明确的里程碑和验收标准
- [ ] 所有文档格式统一、层次清晰
