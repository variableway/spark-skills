# PRD (Product Requirement Document) Writer Skill

> 帮助用户撰写高质量的原始需求文档，使其能够被高效地转化为技术分析和设计方案

## 用途

本 Skill 用于将产品想法转化为结构化的需求文档，作为后续技术设计（使用 Project Analysis Skill）的输入。

---

## 输入

- 产品想法/概念描述
- 目标用户的大致描述
- 期望解决的核心问题
- 任何已知的约束或偏好

## 输出

结构化的需求文档，包含以下 6 个维度：

```markdown
tasks/[project-name]/
└── requirement.md          # 完整的需求文档
    ├── 目标用户            # 详细的用户画像
    ├── 核心目标            # 项目目标和成功标准
    ├── 功能范围            # MVP + Phase 2 + 明确不做
    ├── 约束条件            # 时间/技术/资源约束
    ├── 参考与对标          # 类似产品分析
    └── 输出要求            # 期望的设计文档类型
```

---

## 执行步骤

### Step 1: 提取和细化用户画像

**目标**：将模糊的用户描述转化为具体的、可指导 UI/UX 决策的用户画像

**操作**：
- 询问/推断用户的技能水平
- 明确使用场景和痛点
- 定义设备环境和技术背景

**输出模板**：

```markdown
## 目标用户

### 主要用户

| 属性 | 描述 |
|------|------|
| 技能水平 | [完全零基础 / 有基础概念 / 能写简单脚本 / 专业开发者] |
| 使用场景 | [学习 / 工作提效 / 娱乐 / 生产环境] |
| 痛点描述 | 目前[具体问题]，导致[什么后果] |
| 设备环境 | [Mac/Windows/Linux]、[是否有管理员权限] |

### 用户分层（如适用）

| 用户类型 | 占比 | 特征 | 特殊需求 |
|----------|------|------|----------|
| 类型A | 70% | [描述] | [需求] |
| 类型B | 30% | [描述] | [需求] |
```

**质量标准**：用户画像具体到可以用这些信息做 UI 决策（如：字体大小、术语使用、交互复杂度）

---

### Step 2: 定义核心目标和成功标准

**目标**：明确项目的核心价值和衡量成功的指标

**操作**：
- 用一句话概括产品
- 定义 2-4 个可量化的成功指标

**输出模板**：

```markdown
## 核心目标

用户在[什么场景]下，通过[什么方式]，达成[什么结果]。

### 一句话描述

[产品名] 是一个[产品类型]，帮助[目标用户]解决[核心问题]，通过[核心方法/特性]。

### 成功标准（如何衡量项目成功）

| 指标 | 目标值 | 测量方式 | 优先级 |
|------|--------|----------|--------|
| 指标1 | [数值] | [如何测量] | P0 |
| 指标2 | [数值] | [如何测量] | P1 |
```

**质量标准**：成功标准必须可测量，且能判断是否值得投入资源

---

### Step 3: 划分功能范围（MVP vs Phase 2 vs 不做）

**目标**：明确功能边界，避免范围蔓延

**操作**：
- 列出所有可能的功能
- 分类为 MVP、Phase 2、明确不做
- 为每个 MVP 功能定义验收标准

**输出模板**：

```markdown
## 功能范围

### MVP 必须有的功能

| ID | 功能 | 解决什么问题 | 验收标准 | 估算工时 |
|----|------|--------------|----------|----------|
| F1 | [功能名] | [问题] | [可测试的标准] | [X天] |

### Phase 2 期望有的功能

| ID | 功能 | 解决什么问题 | 优先级 |
|----|------|--------------|--------|
| F2 | [功能名] | [问题] | P1 |

### 明确不做的功能

| 功能 | 不做原因 | 备注（如果以后要做，需要满足什么条件） |
|------|----------|----------------------------------------|
| [功能] | [原因] | [条件] |
```

**质量标准**：MVP 功能要具体到可以转化为 Feature 开发文档

---

### Step 4: 明确约束条件

**目标**：识别所有可能影响设计和实现的约束

**操作**：
- 询问时间要求
- 询问技术要求或偏好
- 询问资源限制

**输出模板**：

```markdown
## 约束条件

### 时间约束

| 类型 | 时间 | 说明 |
|------|------|------|
| 期望交付 | [日期] | 理想交付时间 |
| 硬截止 | [日期/无] | 不可延后的 deadline |

### 技术约束

| 类型 | 约束 | 说明 |
|------|------|------|
| 必须使用 | [技术] | [原因] |
| 不能使用 | [技术] | [原因] |
| 平台要求 | [平台] | [版本/配置要求] |

### 资源约束

| 类型 | 限制 | 说明 |
|------|------|------|
| 团队规模 | [人数] | [角色] |
| 预算 | [金额/无] | [可支出的费用类型] |

### 其他约束

- [合规要求]
- [性能要求]
- [安全要求]
```

**质量标准**：约束具体到可以在技术选型时排除选项

---

### Step 5: 收集参考与对标

**目标**：通过参考产品帮助理解期望的效果和质量标准

**操作**：
- 找出 2-3 个类似产品
- 分析每个产品的优缺点
- 明确我们想要学习/避免的

**输出模板**：

```markdown
## 参考与对标

### 类似产品分析

| 产品名 | 类型 | 优点 | 缺点 | 我们要学习 | 我们要避免 |
|--------|------|------|------|------------|------------|
| [产品A] | [类型] | [优点] | [缺点] | [点] | [点] |

### 可参考的实现

| 参考 | 来源 | 参考方面 |
|------|------|----------|
| [描述/链接] | [来源] | [具体方面] |

### 不想要的例子

| 产品/设计 | 不想要的原因 | 我们的替代方案 |
|-----------|--------------|----------------|
| [例子] | [原因] | [方案] |
```

**质量标准**：参考足够让 AI/设计师理解期望的"感觉"

---

### Step 6: 明确输出要求

**目标**：告诉下游 Skill（Project Analysis）期望的输出

**操作**：
- 列出期望的设计文档类型
- 定义详细程度
- 说明特殊要求

**输出模板**：

```markdown
## 输出要求

### 期望的交付物

- [ ] 技术分析文档（可行性、选型对比、难点分析）
- [ ] 架构设计文档（包含 ASCII 示意图）
- [ ] 功能分解文档（features/ 目录）
- [ ] 实施计划文档（Phase 划分、任务列表）

### 详细程度

- [ ] 高：文档可直接用于开发，无需进一步细化
- [ ] 中：需要少量细化（如：具体接口定义）
- [ ] 低：概念验证级别，需要大量细化

### 特殊要求

- [ ] 必须包含性能分析和优化建议
- [ ] 必须考虑无障碍设计
- [ ] 必须包含安全分析
- [ ] 其他：[说明]
```

**质量标准**：输出要求明确到下游 Skill 可以按此执行

---

## 质量检查清单

在完成需求文档后，检查以下项目：

### 用户相关
- [ ] 用户画像具体到可以做 UI 决策（字体、术语、复杂度）
- [ ] 区分了主要用户和次要用户（如有）
- [ ] 描述了用户的痛点和现有解决方案的不足

### 功能相关
- [ ] MVP 功能可以转化为开发任务
- [ ] 每个 MVP 功能有明确的验收标准
- [ ] 列出了明确不做的功能及原因

### 约束相关
- [ ] 时间约束明确（期望交付 + 硬截止）
- [ ] 技术约束可以在选型时排除选项
- [ ] 资源约束（团队、预算）已说明

### 参考相关
- [ ] 有 2-3 个类似产品作为参考
- [ ] 明确指出了要学习/避免的点
- [ ] 有不想要的例子作为反面教材

### 输出相关
- [ ] 明确列出了期望的设计文档类型
- [ ] 定义了详细程度
- [ ] 说明了特殊要求（如有）

## 常见错误及修正

| 错误类型 | 错误示例 | 修正示例 |
|----------|----------|----------|
| 用户太泛 | "面向所有开发者" | "Mac 用户，有 Python 基础，想要简化 AI 工具配置" |
| 功能无边界 | "功能越多越好" | "MVP 只包含一键安装和基础教程，不包含高级定制" |
| 无成功标准 | "让用户体验好" | "新用户 5 分钟内完成 setup，成功率 >90%" |
| 无约束 | 完全不提及 | "需要在 2 个月内交付 MVP，团队 2 名开发" |
| 无输出要求 | "看着办" | "需要技术分析、架构图、功能分解、Phase 计划" |
| 过于技术 | "使用 React + Redux" | "Web 应用，考虑响应式设计"（技术选型留给设计阶段）|

```

│                     │      │                         │
│   产品想法           │─────►│   PRD Writer Skill      │

│                     │      │   • 结构化需求           │
└─────────────────────┘      │   • 明确约束             │
                             │   • 定义成功标准         │
                             │                         │
                             └───────────┬─────────────┘
                                         │
                                         ▼
                             ┌─────────────────────────┐

                                         │
                                         ▼
                             ┌─────────────────────────┐
                             │   Project Analysis      │
                             │   Skill                 │
                             │                         │
                             │   • 技术选型             │

                             │   • analysis.md         │
                             │   • architecture.md     │
                             │   • features/*.md       │
                             │   • plan.md             │
```

## 示例

### 输入（产品想法）

```
想做一个帮助小白用户配置本地 AI 环境的工具。
用户不太懂技术，想要一键安装。
有教程教他们怎么用。
想要桌面应用的形式。
```

### 输出（requirement.md）

见 `tasks/mindstorm/local-ai-playgound.md`

### 下一步

将 requirement.md 输入 Project Analysis Skill，生成：
- `analysis.md`
- `architecture.md`
- `features/*.md`
- `plan.md`

见 `tasks/mindstorm/learn-ai-agent.md` 和 `tasks/features/*.md`
